# Weapon Visual Remap Reset

ROCK now treats post-mutation equipped-stack evidence as the native remap authority because the visible weapon tree can stay stale after the equipped inventory data has already been mutated. `AttachModToReference` remains useful as secondary evidence, but it is not sufficient for the workbench path and must not authorize `QueueAttachWeapon`. The chosen approach captures native stack writers and direct `BGSObjectInstanceExtra` OMOD add/remove calls after their original mutation returns, then records the equipped player stack directly before generated collision can request native visual remap.

The 2026-05-12 runtime logs showed that `ApplyChangesFunctor::WriteDataImpl` and `BGSInventoryItem::ModifyModDataFunctor::WriteDataImpl` installed but did not fire for the tested AMCAR/Handmade/N99 workbench workflow. Ghidra mapping showed that FO4VR also mutates object-instance OMOD lists through `BGSObjectInstanceExtra::AddMod` at `0x14003CBA0` and `BGSObjectInstanceExtra::RemoveMod` at `0x14003D020`, reached by both stack functors and direct native object-instance paths. ROCK now observes those low-level boundaries, but it binds each signal to the exact `BGSObjectInstanceExtra` address and still requires the pending scan to prove that the mutated extra belongs to the player's equipped weapon.

This avoids broad equip or visual refresh calls. Inventory equips and game-load baselines can identify collision ownership, but native remap is limited to the post-mutation `workbenchApplyChanges` witness or a future source that proves the same stack boundary.
