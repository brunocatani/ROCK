# Weapon Domain

This module owns generated weapon collision, two-handed support, weapon semantic classification, interaction routing, and weapon visual authority. It keeps firearm/melee interaction policy together while relying on native and hand modules for low-level body operations and hand-frame inputs.

## Equipped Instance vs Visual Source Witnesses

Generated weapon collision now treats the equipped instance and the first-person visual tree as separate witnesses. The equipped instance can report a new modded weapon state before the first-person weapon tree exposes the newly attached mesh; rebuilding from the unchanged visual source would cache stale geometry as if it matched the newest mod list.

The runtime uses the object-instance extra/index-data signature as evidence that the mod list changed, but it still builds colliders only from visible, extractable TriShapes. OMOD/index data is not a geometry source. When the instance witness changes and the generated visual source witness is absent or still matches the cached body bank, ROCK keeps the new instance pending and logs the desync for in-game validation. Stale visible sources keep cached colliders active; missing visual roots keep cached state but disable/unpublish bodies because there is no safe transform drive root.

Broad native refresh calls remain out of scope for this path. `AttachModToReference` is retained only as secondary mutation evidence; the workbench authority is the post-mutation equipped stack observed after a native workbench stack writer returns. Current FO4VR runs have shown `BGSInventoryItem::ModifyModDataFunctor::WriteDataImpl` as the weapon-mod path; `ApplyChangesFunctor::WriteDataImpl` remains covered as the sibling stack-write path. `HandleItemEquip`, `SetEquippedItem`, `RequestLoadAnimationsForWeaponChange`, `QueueUpdate3D`, and arm refreshes touch more state than the collider lifecycle needs. The one allowed native remap is `WeaponVisualRemapRuntime`, which first matches the current equipped candidate against ROCK's pending form, instance-data, and object-instance-extra witness from a `workbenchApplyChanges` source, then queues the verified `TaskQueueInterface::QueueAttachWeapon` task through a borrowed two-pointer `BGSObjectInstance` ABI view and `BGSEquipIndex`; collision still publishes only after the visible source tree changes.

`middleHigh->equippedItems` is preferred because it carries the equipped weapon form, instance-data pointer, and equip index used for the remap task, but it is only read through a guarded helper because FO4VR can expose a transient null equipped-item array during first-person skeleton teardown/rebuild. The runtime snapshots those values as non-owning witnesses only; it must not store copied `BGSObjectInstance` values or construct owning `BGSObjectInstanceT` locals because those carry `BSTSmartPointer` ownership over engine-managed instance data. When that list is stale or unavailable, the runtime also builds fallback candidates from first-person and actor equip data, using the same strict pending witness match before queueing. Missing visual roots stay pending and never request native remap; Pip-Boy weapon swaps and draw/holster attach transitions must be allowed to produce a visible tree before ROCK can safely intervene. Remap attempts for stale visible trees are bounded: acquisition failures back off instead of retrying every frame, queued tasks get an observation window, and repeated no-change results stop for that pending witness until new equip evidence appears.

## First-Person Attach Graph Notes

`f4vr::getWeaponNode()` is a passive F4VR-CommonFramework helper, not a native attach function. It returns the first `NiNode` named `Weapon` found by recursively walking `getPlayer()->firstPersonSkeleton`; the walk does not refresh the graph, call native attach code, consult equip data, or materialize internal nodes. This is the first-person attach parent ROCK currently uses as the dominant weapon drive root, but it is not itself proof that every generated weapon part has been published under that parent.

FO4VR native attach stores a second witness that ROCK can inspect: `ActorEquipData::SlotData::node`. In the unpacked FO4VR executable, `TaskQueueInterface::QueueAttachWeapon` at `0x140DAB8F0` packages task type `0x12`; the task handler at `0x140DB0E00` calls virtual slot `0xA5`, which resolves through `PlayerCharacter::AttachWeapon` at `0x140F0A9F0`, `Actor::AttachWeapon` at `0x140DC8190`, and `TESObjectREFR::AttachWeapon` at `0x1403F1E50`. The base attach path reaches the equip visual writer at `0x1401C8150` and visual package builder at `0x1401C9F70`. That writer stores the generated visual package root at `equipData + 0x10 + slotIndex * 0x58 + 0x30`, matching the local `ActorEquipData::SlotData::node` field.

The native visual package is named from `Weapon %s (%08X)` and then attached under native lookup targets named `Weapon` or `WeaponLeft`. The engine uses `BSUtilities::GetObjectByName` at `0x141D137A0`, which is broader than ROCK's local `findNode`: it checks internal named-object data, handles indexed NiNode children, and can fall back to virtual object-name lookup depending on its flags. A collider-source audit should therefore compare the local `firstPersonSkeleton:Weapon` node, native `GetObjectByName(firstPersonSkeleton, "Weapon", true, true)`, matching `SlotData::node`, `PlayerNodes.primaryWeapontoWeaponNode`, and `PlayerNodes.primaryWeaponOffsetNode` before deciding whether a missing part is absent from the engine visual package or only absent from the parent tree ROCK currently scans.
